The situation & what we heard

A manual, agent-only process that can't scale

  • Booking is internal & agent-only — clients phone/email; agents do the work.
  • Approvals & invoicing are manual — slow and error-prone.
  • No unified concept of a multi-segment trip.
  • Growth = more agent hours per booking.

What the clarification call told us
  • #1 need: a complete self-service request → approval flow, connected to the booking system.
  • Goal is efficiency & growth, not selling software.
  • Approvals: 1–3 configurable levels (cost, destination, seniority, cabin class).
  • Reuse the existing booking APIs (search / book / change / cancel); no invoicing API → Microsoft Business Central.
  • 6 roles + Key Travel Agent · pilot first · SSO/MFA login & data privacy · launch fast.

Vision & principles3 / 10
Who we're building for

Six roles across the client & Key Travel

Traveller
Requests a trip that follows the rules; tracks its status.
Travel Arranger
Books on behalf of colleagues.
Approver (levels 1–3)
Approves requests, with the rule context in front of them.
Organisation Administrator
Manages their own people (add/remove staff, set roles).
Finance User
Handles one central invoice; sends figures to Microsoft Business Central.
Key Travel Agent
Sets up & supports organisations; handles exceptions (deeper act-on-behalf comes later).

Each role has its own end-to-end flow — you can switch role and organisation in the prototype's top bar.

What we'll build · MVP vs later

What's in the MVP — and what waits

MVP
  • Organisations, people & roles; book on behalf of others
  • Key Travel-assisted onboarding (manual setup for pilot orgs); accounts via single sign-on (SSO)
  • Travel request form + one unified trip + status
  • Travel rules (limits, destinations, cabin class, department, seniority) with warn & justify
  • 1–3 levels of approval + an approver to-do list
  • Link to the existing booking system + agent hand-off
  • Central invoicing + company card + Business Central
  • Secure login (SSO/MFA), data privacy, activity log
  • Web app — usable on desktop; mobile/tablet-optimized layouts a fast-follow
LATER
  • Key Travel Agent act-on-behalf across organisations
  • Organisations configuring their own rules
  • Automatic sync with an HR system
  • Preview/test approval chains · reporting dashboards & charts
  • Connecting directly to airlines/hotels
  • Traveller safety tracking (duty-of-care)
  • Automatic train & bus booking
  • Individuals paying themselves · a native mobile app (iOS/Android)

  • A web app (usable on desktop; mobile/tablet layouts a fast-follow) plus behind-the-scenes services; a native mobile app comes later.
  • A safe "translator" layer reuses the existing booking APIs (search / book / change / cancel) and shields the new platform from the old system's quirks.

  • Built for many organisations on one platform — from a pilot to hundreds.
#1 unknown

The existing APIs and documentation are incomplete, and there's no invoicing API. We investigate this first, in the discovery week — before finalising the plan.

How it fits together7 / 10
Getting organisations & people started

Onboarding & support

Organisations — set up by hand
  • For the pilot, Key Travel sets up each organisation manually (structure, rules, login, Business Central) — not a built onboarding product.
  • A welcome email at go-live gives the admin a ready-to-use workspace nice-to-have.
  • later a self-service onboarding flow so organisations set themselves up.
Staff — nothing for client IT to do
  • Accounts created automatically on first login via single sign-on (SSO), as a traveller.
  • Admin invites people, or uploads a staff list → invitation emails.
  • later automatic sync with the HR system.
Who manages what
  • The organisation's admin manages their own people.
  • Rules, approvals, budgets & system connections are set up by Key Travel at first.
  • Key Travel provides assisted support during the pilot; deeper act-on-behalf & org self-service come later.
